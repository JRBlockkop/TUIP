                Terminal User Interface Protocol -- TUIP/1.1


2.  Basic Data


2.1  Format

A TUIP message is a structured, line-oriented text document encoded
in UTF-8 and transmitted over TCP.

All lines are terminated by LF (\n). CRLF MAY be accepted by
implementations but MUST NOT be generated by compliant servers.

A TUIP response consists of four logical sections:

    1. Status Line
    2. Headers
    3. Empty Line
    4. Body

General Structure:

    <STATUS LINE>\n
    <HEADER>\n
    <HEADER>\n
    ...
    \n
    <BODY>

Example:

    TUIP 1.1 200
    Content-Length: 124
    Title: Example Page

    t|1,1,15,1>Welcome to TUIP

The empty line between headers and body is mandatory.


2.2  Headers

Headers provide metadata about the response. Each header is transmitted
as a single line in the following format:

    Field-Name: Field-Value

Rules:

- Field names are case-insensitive.
- Field values are case-sensitive unless specified otherwise.
- Whitespace before the value is ignored.
- Each header occupies exactly one line.
- Header folding is NOT allowed.
- Duplicate headers are NOT allowed unless explicitly defined.

Required Headers:
    Title
    Content-Length


Header Definitions:

Title
    Human-readable title for the document.

Content-Length
    Decimal length of the body in bytes (UTF-8 encoded).
    Must match the exact transmitted body length.

2.3  Body

The body contains structured TUIP elements.

The elements are a list.


e|x,y,w,h|p|...>c

e => element name
x => postion on screen
y => postion on screen
w => width on screen
h => height on screen

p => params depends on element

c => Content 



t|1,1,1,1>S

t => text
1 => posx 
1 => posy 
1 => width 
1 => height

S => The Content
